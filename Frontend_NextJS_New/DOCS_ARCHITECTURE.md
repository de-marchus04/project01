# Архитектура и Потоки Данных Платформы (Data Flow & Architecture)

Данный документ описывает иерархию данных, источники истины, икеширование и каналы передачи информации для выхода в production на нагрузку до 100k пользователей.

## 1. Иерархия Системы и Источники Истины (SSOT)

Вся информация на сайте разделена на две большие категории:
1.  **Динамические данные (Контент от Администратора):** Курсы, Статьи, Консультации, Отзывы.
2.  **Пользовательские данные (User Generated Content / State):** Профили, Заказы, Прогресс прохождения.

**База данных (PostgreSQL) — единственный источник истины.**
*Файлы с кодом (.tsx, .ts) полностью освобождаются от "мусора" (жестко зашитых текстов курсов, моков, ссылок)*. Код только *описывает интерфейс*, данные берутся из БД.

---

## 2. Иерархия Сущностей (Entities)

### 👩‍💻 A. Пользователи (Users) & Доступ (Auth)
*   **(Model: `User`)**
    *   **Данные:** Имя, Email, Хэш пароля (Bcrypt), Роль (`USER` / `ADMIN`), Аватар.
    *   **Поток (Login):** Пользователь -> NextAuth (JWT) -> API Route `/api/auth` -> Postgres.
    *   **Поток (Доступ к платформе):** Middleware (`middleware.ts`) перехватывает запрос -> проверяет JWT токен -> если роль не `ADMIN`, отбрасывает от админки.

### 📚 B. Контент (Управляется строго Администратором)
Только администратор имеет право создавать, редактировать или удалять эти сущности.

1.  **Курсы (Courses)**
    *   **Поля:** Заголовок, Описание, Цена, Ссылка на обложку, Видеоуроки.
    *   **Медиа:** Фото и видео загружаются в облако AWS S3 / Cloudinary. В БД хранится только безопасная URL-ссылка на CDN.
2.  **Консультации (Consultations)**
    *   **Поля:** Тип, Длительность, Цена, Описание формата.
3.  **Статьи / Блог (Articles)**
    *   **Поля:** Заголовок, Текст (форматированный/Markdown), Дата публикации, Обложка.

### 🛒 C. Транзакции и Взаимодействие
1.  **Заказы (Orders)**
    *   **Связь:** Пользователь (id) <-> Сущность (Course/Consultation id).
    *   **Статус:** Ожидает оплаты, Оплачено, Отменено.
2.  **Отзывы (Reviews)**
    *   **Связь:** Пользователь (id) -> Отзыв -> Публикация на главной (одобрение админом).

## 3. Цепочки и Каналы передачи контента (Data Pipelines)

*   **Канал 1: Ввод контента (Admin Pipeline)**
    `Admin Next.js Panel` -> Загрузка файла (Облако/S3) -> Получение URL -> Отправка JSON -> `Prisma (ORM)` -> `PostgreSQL`.
*   **Канал 2: Витрина (Public Content Pipeline с ISR)**
    `Пользователь` открывает `/courses` -> Next.js берёт страницу из кэша (HTML) -> *Если кэш устарел, ISR в фоне (через Prisma) тянет новые курсы из PostgreSQL и обновляет HTML для следующих юзеров*. -> Мгновенная отрисовка.
*   **Канал 3: Приватная зона (User Dashboard)**
    `Пользователь` открывает профиль -> Клиентский компонент (отправляет Session Token) -> API проверяет токен -> Prisma тянет список купленных курсов (Orders) -> Рендер.

## 4. План запуска в Production (Масштабирование до 100k)

Для обеспечения надежности и отсутствия зависаний реализована/планируется следующая инфраструктура:

**1. Сервер приложения (Vercel / Hetzner):**
*   *Обоснование:* Next.js App Router (которую мы используем) идеально создана под бессерверные (Serverless) архитектуры Vercel. Распределение статики через глобальный CDN гарантирует, что 90% нагрузки даже не дойдет до вашего сервера базы данных.

**2. База данных (PostgreSQL + Prisma):**
*   *Обоснование:* Prisma создает строгие типы, защищая от ошибок. БД должна быть размещена на управляемом сервисе (Supabase Postgres, AWS RDS или отдельная выделенная VPS), чтобы иметь пулинг соединений (Connection Pooling, например PgBouncer), чтобы 100k людей не открыли 100k соединений к базе одновременно.

**3. Хранилище медиа (AWS S3 / Cloudinary):**
*   *Обоснование:* БД не предназначена для хранения картинок весом 5мб или видео-уроков. Файлы отдаются напрямую с мирового кэша S3, а база хранит лишь ссылки.

**4. Подход к рендерингу (ISR - Incremental Static Regeneration):**
*   *Обоснование:* Вместо запроса к БД при каждом заходе на страницу `app/courses/page.tsx`, мы прописываем `export const revalidate = 3600`. Сервер генерирует статику 1 раз в час. База данных «спит», а пользователи моментально получают ответ.

**5. Мониторинг (Sentry):**
*   *Обоснование:* Устанавливается пакет `@sentry/nextjs`. Если при покупке или просмотре ломается интерфейс — вы первым получаете письмо со строкой кода (StackTrace).
